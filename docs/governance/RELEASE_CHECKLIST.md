## 1. 目标
- 在合入/发布前做统一质量门禁，避免“能跑但不可控”。

## 2. 适用范围
- 所有影响主流程或配置的改动。

## 3. 输入与输出
- 输入：代码改动、PRD/TECH_SPEC、测试结果。
- 输出：发布结论（通过/阻断）与签字记录。

---

## 4. 发布前必检项（全部通过才可发布）
- [ ] PRD 已更新，包含明确验收标准
- [ ] TECH_SPEC 已更新，包含影响面与回退说明
- [ ] 配置变更已在 CONFIG_POLICY 记录
- [ ] Smoke 测试通过
- [ ] Regression 测试通过或有批准豁免
- [ ] `run_summary.json` 已产出并归档
- [ ] 关键失败路径已验证（至少 1 条）
- [ ] 输出格式与命名符合规范
- [ ] 变更记录已写入 CHANGE_POLICY

## 5. 阻断条件
- 缺失任一最小产物（需求/技术/验收）
- 回归退化且无说明
- 配置来源不明确
- 关键路径错误无法解释

## 6. 发布记录模板
- **发布批次**：格式RELEASE-YYYY-XXX，如RELEASE-2026-001
- **负责人**：发布负责人姓名
- **发布时间**：YYYY-MM-DD HH:MM:SS
- **风险级别**：高/中/低
- **回滚预案**：回滚方案
- **结论**：PASS / BLOCK

---

## 7. 发布后验证

### 7.1 验证内容
- **功能验证**：
  - 核心功能是否正常（切片/分类/导出）
  - 新增功能是否按预期工作
  - 修复的Bug是否确实修复
- **性能验证**：
  - 处理速度是否在预期范围内
  - 内存/显存占用是否正常
  - 是否有性能退化
- **准确率验证**：
  - 各标签分类准确率是否稳定
  - 是否有不可解释的退化
- **兼容性验证**：
  - 旧版本输出是否兼容
  - 配置是否向后兼容

### 7.2 验证时间
- **发布后1小时内**：完成功能验证
- **发布后4小时内**：完成性能验证
- **发布后24小时内**：完成准确率验证

### 7.3 验证方法
- **Smoke测试**：快速验证核心功能
- **监控指标**：查看处理时间/内存/显存
- **Regression测试**：对比测试样本集结果
- **人工抽检**：抽查部分输出结果

---

## 8. 回滚流程

### 8.1 回滚触发条件
出现以下任一情况时，需要考虑回滚：
- 核心功能不可用
- 性能严重退化（>30%）
- 准确率严重退化（>20%）
- 数据丢失或损坏
- 安全漏洞
- 用户严重投诉

### 8.2 回滚步骤
1. **步骤1**：停止发布
   - 暂停新的处理任务
   - 保存当前状态
2. **步骤2**：回滚代码
   - Git revert到上一个稳定版本
   - 确认代码回滚正确
3. **步骤3**：回滚配置
   - 恢复配置到上一个稳定版本
   - 确认配置回滚正确
4. **步骤4**：验证
   - 执行Smoke测试
   - 验证核心功能正常
   - 验证性能/准确率恢复
5. **步骤5**：记录
   - 记录回滚原因
   - 记录回滚时间
   - 记录回滚结果

### 8.3 回滚记录模板
| 字段 | 值 |
|-----|-----|
| 回滚ID | ROLLBACK-YYYY-XXX |
| 触发原因 | 为什么回滚 |
| 回滚时间 | YYYY-MM-DD HH:MM:SS |
| 回滚前版本 | 回滚前的版本号/Commit |
| 回滚后版本 | 回滚后的版本号/Commit |
| 验证结果 | PASS/FAIL |
| 回滚人 | 回滚人姓名 |
| 备注 | 其他信息 |

---

## 9. 验收标准
- 每次发布都有完整清单记录。
- 阻断条件能被明确触发并执行阻断。
