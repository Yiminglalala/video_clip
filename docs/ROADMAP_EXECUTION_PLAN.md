## 1. 总目标
- 先建立治理基础设施，再推进效果优化，确保每次迭代“可执行、可验证、可复盘”。

## 2. 适用范围
- 全项目开发与协作流程。
- 当前阶段：切片链路优先；字幕链路不回退。

## 3. 输入与输出
- 输入：PRD 条目、历史问题、运行日志、测试样本。
- 输出：阶段交付物、验收报告、风险清单、下一阶段计划。

## 4. 阶段计划

### Phase A：治理基线（第 1 周）
- **阶段目标**：建立统一文档与执行入口，停止"无规范开发"。
- **里程碑**：M1 - 治理文档v1.0发布
- **量化指标**：
  - 规范文档齐全并可直接用于任务执行
  - 任一新任务能在30分钟内进入执行流
  - 所有核心文档已创建并通过评审
- **任务包**：
  - 建立 `docs/governance/` 规范中心
  - 落地 PRD/TECH_SPEC/TEST_STRATEGY/RELEASE_CHECKLIST 等文档
  - 统一状态标签与最小产物约束
- **优先级**：P0
- **责任角色**：
  - Owner：技术负责人
  - Support：开发/测试
- **输入**：当前代码现状、历史讨论、已有文档
- **输出**：规范文档集 v1
- **完成定义（DoD）**：
  - 规范文档齐全并可直接用于任务执行
  - 任一新任务能在 30 分钟内进入执行流
- **风险**：
  - 文档覆盖不足导致执行歧义
- **回滚策略**：
  - 保留旧流程入口，逐步切换，不中断现有研发

### Phase B：执行流落地（第 2 周）
- **阶段目标**：把文档约束变成日常执行动作。
- **里程碑**：M2 - 首轮流程演练完成
- **量化指标**：
  - 至少1个模拟任务完整走通
  - 发布清单可执行且有记录
  - 团队成员对流程理解度>80%
- **任务包**：
  - 固化"需求 -> 技术 -> 测试 -> 发布"流程
  - 每个任务都产出 `run_summary.json`
  - 建立 issue 模板并试运行 1 轮
- **优先级**：P0
- **责任角色**：
  - Owner：技术负责人
  - Support：开发/测试
- **输入**：规范文档集 v1
- **输出**：首轮流程演练报告
- **完成定义（DoD）**：
  - 至少 1 个模拟任务完整走通
  - 发布清单可执行且有记录
- **风险**：
  - 团队执行习惯未转变
- **回滚策略**：
  - 允许并行老流程 1 周，但新任务强制走新模板

### Phase C：配置与测试治理（第 3 周）
- **阶段目标**：减少隐性风险，提升变更可控性。
- **里程碑**：M3 - 配置与测试基线建立
- **量化指标**：
  - 不再新增阈值硬编码
  - 回归报告可自动对比基线
  - 测试覆盖率>60%
- **任务包**：
  - 硬编码阈值迁移清单梳理与分批迁移
  - 固定测试样本集与回归基线
  - 建立每周性能与准确率报告节奏
- **优先级**：P1
- **责任角色**：
  - Owner：开发负责人
  - Support：测试
- **输入**：执行流落地结果、历史 run_summary
- **输出**：配置迁移清单 + 回归基线报告
- **完成定义（DoD）**：
  - 不再新增阈值硬编码
  - 回归报告可自动对比基线
- **风险**：
  - 历史参数耦合，迁移易引入回归
- **回滚策略**：
  - 参数迁移按小批次，支持逐项回退

### Phase D：效果优化迭代（第 4 周及后续）
- **阶段目标**：在稳定流程下持续提升识曲与分类效果。
- **里程碑**：M4 - 核心效果指标达标
- **量化指标**：
  - 合唱识别准确率>70%
  - SOLO识别准确率>60%
  - 未知歌曲率<10%
  - 指标持续改善，且无不可解释退化
- **任务包**：
  - 合唱/SOLO识别专项
  - 未知歌曲率降低专项
  - 失败可解释性增强
- **优先级**：P1
- **责任角色**：
  - Owner：算法/开发负责人
  - Support：测试
- **输入**：稳定执行流、配置清单、回归基线
- **输出**：专项优化报告与版本发布记录
- **完成定义（DoD）**：
  - 指标持续改善，且无不可解释退化
- **风险**：
  - 追求效果导致流程绕过
- **回滚策略**：
  - 任何效果改动都必须附带最小产物，否则不发布

## 5. 每周例行产物
- 周任务板（状态标签）
- 周测试报告（含 run_summary 汇总）
- 周风险清单（含阻断项）
- 周变更记录（CHANGE_POLICY）

## 6. 验收标准
- 阶段输出均可落地执行，不依赖口头约定。
- 所有后续开发都能严格套入该执行总纲。
