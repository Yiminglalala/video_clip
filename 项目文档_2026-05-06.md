## 一、项目概述

### 1.1 项目定位

这是一个面向演唱会视频的本地 Web 工具，当前主应用由 Streamlit 提供双 Tab：

1. **视频切片**：分析演唱会视频结构，生成可预览、可编辑、可导出的短视频片段。
2. **字幕生成**：使用豆包语音 API 识别歌词/语音，生成横排整句字幕并烧录到视频。

当前项目重点已经从“自动听歌识曲”收敛到“稳定切片 + 可编辑标签 + 豆包字幕 + 统一输出规格”。听歌识曲相关代码仍有遗留模块，但主切片链路默认禁用，不作为当前产品主路径。

### 1.2 当前核心能力

- **双 Tab 应用**：`🎬 视频切片` 在前，`🎤 字幕生成` 在后。
- **切片四步工作流**：选择视频 → 结构分析 → 预览编辑 → 导出视频。
- **标签可编辑**：Step 3 可修改输出标签和 SongFormer 标签，也可调整片段起止时间。
- **字幕简化主链路**：字幕页只走豆包 API + ASS 硬字幕烧录，固定按句横排显示。
- **统一输出规格**：导出固定 MP4，H.264 + AAC，60fps，`yuv420p`。
- **统一分辨率策略**：竖屏输入输出 `1080x1920`；横屏输入可选 `1920x1080`、`1080x1440`、`1080x1920`。
- **画面适配方式**：统一使用铺满画布后居中裁切，不补黑边，不拉伸。
- **GPU 优先**：SongFormer、Demucs、FFmpeg NVENC 优先使用 GPU，并有 CPU fallback 或显式告警。

## 二、当前技术架构

### 2.1 总体架构

```text
Streamlit Web UI (app.py)
├─ Tab 1: 视频切片
│  ├─ Step 1 选择视频与参数
│  ├─ Step 2 结构分析
│  │  ├─ FFmpeg 提取音频
│  │  ├─ OCR / 音频 / MSAF 边界辅助
│  │  ├─ SongFormer 段落分析
│  │  ├─ 声学特征二次修正
│  │  └─ 可选豆包 ASR 缓存字幕
│  ├─ Step 3 切片预览与标签编辑
│  └─ Step 4 导出 MP4 切片
│
└─ Tab 2: 字幕生成
   ├─ 上传或选择本地视频
   ├─ 提取音频 + 可选预处理
   ├─ 豆包 API 识别
   ├─ ASS 字幕生成
   └─ FFmpeg 烧录字幕输出 MP4
```

### 2.2 当前技术栈

| 层级 | 技术 | 当前用途 |
|---|---|---|
| Web UI | Streamlit | 双 Tab 页面、参数配置、预览、导出 |
| 视频处理 | FFmpeg | 音频提取、切片、字幕烧录、转码 |
| ASR | 豆包语音 API | 字幕页主识别引擎；切片页可选字幕缓存 |
| 音频分析 | librosa / scipy / soundfile | 音频特征、预处理、辅助分类 |
| 音乐结构 | SongFormer | 当前切片主分类模型 |
| 声学修正 | `SegmentClassifier` | 修正 SongFormer 对副歌/主歌等标签的偏差 |
| 输出策略 | `src/output_spec.py` | 分辨率、字体、裁切、FFmpeg 公共参数 |
| GPU | CUDA / PyTorch / NVENC | SongFormer、Demucs、视频编码加速 |
| 数据存储 | SQLite / JSON | 样本库雏形、配置持久化 |

## 四、核心功能现状

### 4.1 视频切片页

当前切片页是四步工作流：

1. **选择视频**
   - 支持上传或粘贴本地路径。
   - 输入格式不限制，能被 FFmpeg 解码即可。
   - 输出固定 MP4。
   - 高级参数包括最小时长、最大时长、切片模式、是否生成字幕、横屏输出分辨率。

2. **结构分析**
   - 检测 CUDA 状态。
   - 强制启用 SongFormer，`strict_songformer=True`。
   - SongFormer 缺依赖时直接提示安装，不静默降级。
   - 分析结果保存在 `st.session_state.slice_analysis_result`、`slice_export_segments`、`slice_segments`。

3. **预览编辑**
   - 当前只预览选中的切片，不再展示整条视频。
   - 可编辑：
     - 输出标签：`主歌 / 副歌 / 合唱 / 讲话串场 / 乐器SOLO`
     - SongFormer 标签：内部英文标签
     - 开始时间、结束时间
   - 手动修改会同步到导出阶段。

4. **导出视频**
   - 基于 Step 3 的最终编辑结果导出。
   - 文件名格式：`YYYYMMDD_歌曲名_序号_类型.mp4`。
   - 当前歌名默认多为 `未知歌曲`，除非切片字幕启用且歌词匹配成功。

### 4.2 字幕生成页

当前字幕页是简化版：

1. 上传视频或粘贴本地路径。
2. 提取音频。
3. 可选音频预处理。
4. 调用豆包 API。
5. 生成 ASS 字幕。
6. 使用 FFmpeg 烧录到 MP4。

当前字幕策略：

- 字幕模式固定为 `sentence`，不再逐字显示。
- 豆包 API `words_per_line` 默认 15。
- 本地 ASS 生成时超过 14 个中文字符会拆成多句顺序展示，避免自动换行。
- 字幕字体来自 `src/output_spec.py`：`ZiYuWenYunTi`。
- 字幕字号当前为 90。
- ASS `MarginV` 当前为 240。
- 字幕为硬字幕，导出视频里可直接看到，但不能像字幕轨一样无损修改文字。

### 4.3 输出规格

输出规格集中在 `src/output_spec.py`：

| 场景 | 输出 |
|---|---|
| 竖屏输入 | `1080x1920` |
| 横屏输入选 `1920x1080` | `1920x1080` |
| 横屏输入选 `1080x1440` | `1080x1440` |
| 横屏输入选 `1080x1920` | `1080x1920` |

统一画面滤镜：

```text
scale=目标宽:目标高:force_original_aspect_ratio=increase:flags=lanczos,
crop=目标宽:目标高:(in_w-目标宽)/2:(in_h-目标高)/2,
format=yuv420p
```

统一媒体参数：

- 容器：MP4
- 视频：H.264
- 音频：AAC
- 帧率：60fps
- 像素格式：`yuv420p`
- 色彩参数：BT.709 / TV range
- 编码优先级：`h264_nvenc` → `libx264`

### 4.4 歌曲识别现状

当前主切片处理器中：

```python
self._disable_song_identify = True
self._disable_lyrics_identify = True
```

因此 ACRCloud / Shazam 这类听歌识曲不再是当前切片主路径。`lyric_subtitle.py` 中仍保留 ACRCloud、ShazamIO、ShazamAPI 等识曲辅助代码，但当前主 UI 没有暴露识曲配置，切片页 Step 1 也明确是“无识曲配置”。

如果切片页启用“生成字幕”，系统会对歌曲音频调用豆包 ASR，后续可能尝试通过歌词和歌手名匹配歌名；匹配失败时回退为 `未知歌曲`。

## 五、关键模块说明

### 5.1 `app.py`

职责：

- Streamlit 双 Tab 主应用。
- 管理切片四步流程。
- 管理字幕页豆包识别流程。
- 管理 `slice_config.json` 持久化。
- 负责 Step 3 手动编辑数据到 Step 4 导出数据的同步。

当前注意点：

- 豆包 AppID 和 Access Token 仍在代码默认值中，后续建议迁移到环境变量或本地私有配置。
- 切片页 `enable_subtitle` 默认取 `slice_config.json`，源码默认是 `False`，当前本地配置可能不同。

### 5.2 `src/processor.py`

职责：

- 视频分析主编排。
- SongFormer 段落分析。
- 声学特征二次修正。
- 安全边界处理。
- 导出片段队列构建。
- 可选豆包字幕缓存与导出烧录。

当前重要行为：

- `ProcessingConfig` 已包含 `landscape_resolution_choice` 和 `source_orientation`。
- SongFormer 默认为严格模式，失败时会报错。
- 处理器有 GPU 任务锁和 GPU 清理逻辑。
- 导出文件名固定为 `日期_歌曲名_序号_类型.mp4`。

### 5.3 `src/ffmpeg_processor.py`

职责：

- FFmpeg 切片和转码。
- 统一输出 MP4。
- 统一重编码，不再依赖 copy 输出。
- 结束边界增加安全边距，避免最后帧带入下一片段内容。

当前重要行为：

- `cut_video()` 开头强制输出后缀为 `.mp4`。
- 即使 UI 选择 fast，当前主分支实际也会走统一重编码输出路径。
- 优先 NVENC，失败后回退 libx264。

### 5.4 `src/output_spec.py`

职责：

- 输出分辨率策略。
- 字幕字体配置。
- FFmpeg 裁切滤镜。
- ASS 字体目录处理。

当前配置：

```python
LANDSCAPE_RESOLUTION_CHOICES = ("1920x1080", "1080x1440", "1080x1920")
DEFAULT_LANDSCAPE_RESOLUTION = "1920x1080"
SUBTITLE_FONT_FAMILY = "ZiYuWenYunTi"
```

### 5.5 `src/doubao_api.py`

职责：

- 调用豆包语音 API。
- 轮询识别结果。
- 将豆包结果转换为项目内部 `words / sentences / text` 格式。
- 合并过碎的句子片段。

当前行为：

- `recognize()` 默认 `words_per_line=15`、`max_lines=1`。
- `format_result()` 返回 `engine="doubao"`。

### 5.6 `src/sample_library.py` 与 `src/auto_optimizer.py`

当前状态：

- 已有 SQLite 样本库和优化历史表雏形。
- 已有 `VideoSample / EditableSegment / OptimizationHistory` 数据模型。
- `src/ui.py` 中有样本库和参数优化演示页。
- 但这些能力尚未接入当前 `app.py` 切片主流程。
- `auto_optimizer.py` 目前更偏统计和占位逻辑，还没有真实回放分类并生成可应用规则。

后续如果要做“标签库 + 手动触发校准”，应该复用这些模块但升级数据结构，不建议从零重写。

## 七、使用流程

### 7.1 视频切片

1. 打开 `http://localhost:8501`。
2. 进入 `🎬 视频切片`。
3. Step 1 选择视频：
   - 上传视频或粘贴本地路径。
   - 设置最小时长、最大时长。
   - 选择切片模式。
   - 选择是否生成字幕。
   - 横屏视频选择输出分辨率。
4. Step 2 点击开始分析。
5. Step 3 选择切片进行预览和编辑：
   - 可改输出标签。
   - 可改 SongFormer 标签。
   - 可改开始/结束时间。

### 7.2 字幕生成

1. 进入 `🎤 字幕生成`。
2. 上传视频或粘贴本地路径。
3. 横屏视频选择输出分辨率。
4. 点击开始处理。
5. 系统调用豆包 API 识别并烧录字幕。
6. 页面展示输出视频和下载按钮。

## 八、当前状态与差异修正

### 8.1 与旧文档不一致的地方

| 旧文档描述 | 当前代码实际状态 | 修正结论 |
|---|---|---|
| 字幕支持豆包、Whisper、FunASR 多引擎主路径 | 当前 UI 主路径只使用豆包 API；Whisper 主要保留在旧模块 fallback 中 | 文档改为“豆包为主” |
| 自动听歌识曲是核心功能 | 当前切片主处理器默认禁用听歌识曲，UI 无 ACR/Shazam 配置 | 文档改为“识曲非主路径” |
| fast 模式是流复制 | 当前 `cut_video()` 主路径统一重编码并输出标准 MP4 | 文档改为“mode 保留兼容，输出统一重编码” |
| AST 场景分类未接入 | `classifier.py` 未接入，但 `audio_analyzer.SegmentClassifier` 已用于声学修正 | 文档改为“独立 classifier.py 非主路径，声学修正已存在” |
| 字幕自动换行 | 当前策略是豆包请求 15 字，本地超过 14 字拆句，避免换行 | 文档改为“长句拆成多句” |
| 歌曲识别可用但待优化 | 当前默认回退 `未知歌曲`，识曲相关为遗留/辅助代码 | 文档改为“歌名识别非当前稳定能力” |

|---|---|---|

### 8.3 当前已知问题与风险

1. **样本库/标签库尚未接入主流程**
   - `sample_library.py` 和 `auto_optimizer.py` 存在，但当前 `app.py` 没有使用。
   - 参数优化目前不能真实影响下一次切片结果。

   - `lyric_subtitle.py` 仍保留 ACRCloud/Shazam 代码。

   - `requirements.txt` 只是最小依赖。
   - SongFormer、Demucs、PyTorch、MSAF 等依赖未完整固化。

4. **配置来源仍需收敛**
   - `slice_config.json` 会记录用户上次选择。

   - `processor.py` 中存在一些 legacy 路径和重复导出逻辑，需要分阶段清理。

7. **部分源码注释存在编码不一致风险**
   - PowerShell 读取部分文件注释时会出现乱码。
   - 不一定影响运行，但会影响长期维护和文档提取。

## 九、下一步优化建议

### P0：标签库与手动校准闭环

目标：让用户修改后的标签反哺下一次切片。

建议做法：

1. 在 Step 3 增加“保存当前片段到标签库”和“保存全部修改到标签库”。
2. 保存原始 SF 标签、最终 SF 标签、原始输出标签、最终输出标签、时间段、特征、视频来源。
3. 手动点击“优化标签规则”生成报告。
4. 用户确认后写入 `data/learned_label_rules.json`。
5. `processor.py` 在 SongFormer 之后读取规则做二次校准。

第一阶段只做后处理规则校准，不训练 SongFormer 本体。

### P1：配置与依赖治理（已启动）

1. 将豆包凭证迁移到环境变量或本地私有配置。
2. 补充完整依赖说明，区分“最小 UI 依赖”和“完整 GPU 切片依赖”。
3. 给 `slice_config.json` 增加字段说明和默认值策略。
4. 明确 `8501` 稳定环境、`8502` worktree 测试环境。

当前已落地：

- 新增 `src/runtime_config.py`，统一读取环境变量和 `local_config.json`。
- 新增 `local_config.example.json`。
- `local_config.json`、`.env` 已加入 `.gitignore`。
- 新增 `docs/CONFIGURATION.md` 和 `docs/DEPENDENCIES.md`。
- 豆包默认凭证已从 `app.py` 和 `processor.py` 移除。

### P2：旧代码清理（安全版已启动）

1. 标记或移除非主路径的 ACRCloud/Shazam 识曲逻辑。
2. 清理 `src/ui.py` 旧演示 UI，或将其能力迁移到 `app.py`。
3. 梳理 `processor.py` 中重复导出函数和 legacy fallback。
4. 修复源码注释编码不一致问题。

当前已落地：

- `src/lyric_subtitle.py` 的 legacy 听歌识曲默认禁用，只有设置 `ENABLE_LEGACY_SONG_IDENTIFY=1` 才会启用。
- `src/ui.py` 已标记为旧版原型 UI，并在运行时提示当前正式入口是 `app.py`。
- 新增 `docs/LEGACY_MODULES.md`，列出主路径和 legacy 模块边界。

暂未直接执行的高风险项：

- `processor.py` 大规模拆分与重复逻辑删除：需要单独 worktree 和完整 E2E 回归后再做。
- 全文件注释编码修复：当前部分文件已有历史乱码，批量转码可能误伤有效代码，建议单独做只读 diff + 编译验证。

### P3：量化测试体系

1. 建立固定测试视频集。
2. 生成 `run_summary.json`。
3. 验证导出规格：MP4/H.264/AAC/60fps/yuv420p/分辨率。
4. 验证标签准确率、片段重叠率、片段时长分布、字幕可见性。
